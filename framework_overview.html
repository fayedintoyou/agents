Pipeline-Gated Agentic Workflows

Functional phase decomposition over persona-based parallelism — aligned with Anthropic's own harness research
Core thesis: Gate on irreversibility, not difficulty. Structure work as a pipeline of functionally distinct phases with human checkpoints at natural decision boundaries. Separate the agent that does the work from the agent that judges it. Parallelize exploration, not implementation.

The Pipeline

Five phases. Each is an entry point — users start at whatever level matches their current clarity. Phase boundaries are context resets by design: each phase consumes a structured handoff artifact from the previous phase with a clean context window, avoiding the coherence loss and "context anxiety" that degrades long-running single-context sessions.

no gate
/research
Explore the landscape. What exists, what are the options? This is where parallelism lives — multiple workers can explore different solution spaces simultaneously.
review
/design
Given an intent, produce an approach. Multiple options with tradeoffs. Output: design artifact with recommended path.
approve
/scope
Break a design into implementable work chunks. Output: discrete stories with acceptance criteria. (Unique to this framework — Anthropic's harness skips this step.)
no gate
/do
Execute a specific work chunk. Write code, run tests. Single-agent, sequential. Commits non-blocking to feature branch.
auto/gate
/eval
Separate agent reviews /do output against acceptance criteria from /scope. Adversarial, checklist-driven. Escalates or auto-approves based on trust level.

Handoff artifacts between phases serve double duty: they're the human review surface and the clean-slate context for the next agent.

Alignment with Anthropic's Harness Research

This framework isn't competing with Anthropic's multi-agent architecture — it's converging on the same principles from a developer-workflow angle. Anthropic's three-agent harness (planner → generator → evaluator) and their research system both validate the same decomposition axis: functional phase, not persona.

Anthropic: Planner
/research + /design + /scope
Anthropic collapses planning into one agent. We decompose it into three entry points because developer workflows need different levels of human involvement at each stage — sometimes you already know what to build and just need scoping.
Anthropic: Generator
/do
Direct equivalent. Single agent, focused execution. Anthropic found that context resets between generation rounds beat compaction — our phase boundaries achieve the same thing structurally.
Anthropic: Evaluator
/eval
The key insight both architectures share: agents reliably over-praise their own work. Separating generation from evaluation is a "strong lever." Our /eval makes this a first-class pipeline stage rather than a harness-level concern.
Anthropic: Research subagents
/research (parallel workers)
Anthropic's research system spawns parallel subagents for exploration. Same principle: parallelism pays off for information gathering, not for implementation. Token usage explains 80% of research performance — spend tokens exploring, not reviewing.
Where we add to Anthropic's model

The /scope phase has no equivalent in Anthropic's harness. They go from planning to generation directly. For real-world feature development, the gap between "here's the approach" and "start coding" is where things fall apart — a design that works conceptually might be five stories or fifteen, might have hidden dependencies, might need sequencing. /scope is the bridge that makes /do tractable.

The Evaluator Gap (Why /eval Exists)

The problem

Agents asked to evaluate their own work consistently over-praise it. This is especially bad for subjective work (does this UI feel right?) but persists even for objectively verifiable tasks. Without a separate evaluation pass, the only check is the human at the gate — which blocks the path to higher trust levels.

The solution

/eval is a separate agent with a fresh context that receives the /do output and the acceptance criteria from /scope. It runs checklist-driven verification (not persona roleplay) and one adversarial pass ("try to break this"). At low trust, it presents findings to the human. At high trust, it auto-approves passing work and only escalates failures.

Why this unlocks trust scaling

Without /eval, collapsing human gates means no one is checking the work. With /eval, collapsing human gates means the automated check is always running — humans only need to intervene when /eval flags something. This is the mechanism that makes trust_level: high safe rather than reckless.

Where Parallelism Belongs (and Doesn't)

Parallelize: /research

Spawn multiple workers exploring different solution spaces simultaneously. This is where token spend scales linearly with quality — more exploration, more options, better decisions. Anthropic's data confirms: multi-agent parallelism excels at tasks involving heavy parallelization and information that exceeds single context windows.

Don't parallelize: /do

Implementation has too many dependencies between steps. Parallel coding agents step on each other — merge conflicts, contradictory patterns, duplicated work. Keep /do single-agent and sequential. This is where the persona-director approach fails hardest: 8 agents "implementing" the same feature produces convergence theater, not better code.

Handoff Artifacts

Phase boundaries aren't just process gates — they're context engineering. Each handoff artifact serves two audiences simultaneously:

For the human
The review surface. A structured document that makes the gate decision easy: is this the right direction? Does this scope make sense? Did this implementation meet acceptance criteria? The artifact is what gets approved or rejected.
For the next agent
A clean-slate context. No accumulated conversation history, no compaction artifacts, no context anxiety. Just a well-structured document with exactly the state needed to begin the next phase. This is why phase boundaries are a feature: each agent starts fresh with maximum effective context.

When personas do add value

If you want multi-perspective review, use checklist-driven review passes within /eval rather than character sheets. Not "be a security expert" but "check for: unsanitized inputs, exposed secrets, missing auth, injection vectors." Concrete criteria outperform role-play prompts. Pair with one adversarial pass: "try to break this, find the case that makes it fail."

HITL Gating by Domain

Gate frequency should be inversely proportional to how well automated checks can verify correctness.

Backend / API
High automated verifiability. Tests, type checks, contract validation catch most issues. /eval can auto-approve when tests pass. Gate at the PR level. Tighter gate only for migrations or schema changes (hard to reverse).
Frontend / UI
Low automated verifiability today. Code can be "correct" and still look wrong. Gate at visual milestones: layout → styling → interactions. Collapses as visual self-check tooling improves (screenshot comparison, design spec diffing).
Infrastructure
High irreversibility. DNS, IAM, networking changes are hard to undo. Gate before apply, even with plan previews. "terraform plan" is the gate, not the substitute for one.
Data / Migrations
Highest irreversibility. Destructive migrations can't be undone. Always gate. Require dry-run output and rollback plan before approval.

Trust Levels (Configurable per Skill)

Don't rewrite skills as models improve — turn a dial. Each level defines which gates are active and how /eval behaves.

trust_level: low
Gate after every phase. Research → review → Design → approve → Scope → approve → Do → Eval presents findings to human → human approves. Maximum human involvement. /eval always escalates. Use for unfamiliar domains, high-stakes systems, or early trust-building with a new model.
trust_level: medium
Gate after design and after eval. Research flows into design automatically. Scope flows into do automatically. /eval runs automatically and presents pass/fail to human with findings. Human decides the approach and validates the result. Most teams should live here.
trust_level: high
Gate only before irreversible actions — merge, deploy, send, delete. The full pipeline including /eval runs autonomously. /eval auto-approves passing work and only escalates failures to human. Use when test coverage is high and the blast radius is contained.

On Commit Gating (Drop It)

Principle

Commits are an implementation detail, not a meaningful unit of review. Some commits are "renamed a variable" and some are "rewrote the auth flow." Gating on commits treats them as equivalent. They aren't.

Instead

Gate on outcomes — "the page now renders with real data" or "the API handles the new payload format." These are the natural decision boundaries where human judgment adds value. /eval checks outcomes against /scope acceptance criteria, not commit diffs.

Best of Both Worlds

Use non-blocking commits. The agent commits frequently to a feature branch. The human can watch the commit stream asynchronously and intervene at any point ("stop, wrong direction"), but only /eval → human approval is a blocking gate. Full audit trail, zero friction.

Future-Proofing

The Trend

Gates move outward over time. Today you gate between design and implementation. Tomorrow you gate between "here's a fully implemented PR that /eval approved" and "merge." The interior checkpoints collapse as model capability and self-verification improve. Design your skills so this collapse happens by changing trust_level, not rewriting the skill.

Key signals that a gate can be removed:

Visual self-check

When the agent can reliably screenshot its own work and compare to a spec, frontend visual gates collapse. Add a visual_self_check: true flag — /eval screenshots the result and only escalates when something looks off.

Test-driven confidence

When test coverage is high enough that passing tests genuinely means correct behavior, /eval can auto-approve backend work. Human gates collapse to merge-only.

Async optimistic execution

The agent presents a design and continues implementing optimistically. Human reviews the design while code is being written. If approved, you're ahead. If rejected, discard. Only works when cost of discarding is low — which is increasingly true.
